## Issues Found & Fixes Applied

### Issue #1 — CRITICAL: `users` table missing from dev database

**Symptom:** API server logged `ERROR: relation "users" does not exist` on every startup (super user seed failure) and on every `POST /api/auth/login` attempt (HTTP 500). Login was completely broken.

**Root Cause:** The `users` schema was defined in code (`lib/db/src/schema/users.ts`) but had never been pushed to the database. The project uses `drizzle-kit push` (not migration files) for schema sync, and this push had not been run after the `users` table was added.

**Fix:** Ran `pnpm --filter @workspace/db run push-force` to sync the Drizzle schema to the dev database.

**Verification:**
- Dev database now shows 5 tables: `injections`, `tenants`, `tiers`, `users`, `webhook_events`.
- API server restarted — super user seeded successfully (`abc17@gmail.com`, role `superuser`).
- Login endpoint operational.

### Issue #3 — MODERATE: 5 type errors in frontend (`eng-architect`)

**Symptom:** `pnpm run typecheck` failed on `@workspace/eng-architect` with 5 instances of:
```
error TS2339: Property 'error' does not exist on type 'ErrorType<ApiError>'.
```

**Root Cause:** The `onError` callbacks in mutation hooks referenced `err.error`, but the error type (`ApiError`) extends JavaScript's `Error` class, which has a `.message` property — not `.error`. This was also a runtime bug: error toasts would have always shown the fallback text (e.g., "An error occurred") instead of the actual API error message from the server.

**Fix:** Changed `err.error` to `err.message` in all 5 call sites across 4 files:

| File | Instances Fixed |
|---|---|
| `src/components/InjectComposerDialog.tsx` (line 60) | 1 |
| `src/pages/Injections.tsx` (line 56) | 1 |
| `src/pages/TenantDetail.tsx` (lines 88, 105) | 2 |
| `src/pages/Tenants.tsx` (line 76) | 1 |

**Verification:** Full workspace typecheck (`pnpm run typecheck`) passes with zero errors across all packages.

### Issue #4 — CRITICAL: Production deployment stuck at Provisioning

**Symptom:** Publishing held indefinitely at the "Provisioning" stage.

**Root Cause:** The production database (separate from dev) did not have the `users` table. The API server's production build step (`artifact.toml`) only ran esbuild — there was no schema sync step. When the production server started, it attempted to seed the super user and query the `users` table, which didn't exist. The production `DATABASE_URL` at the time of the failed publish was pointing to a database without the `users` table, causing the server to error on startup and fail the health check (`/api/healthz`), which kept the deployment stuck in Provisioning.

**Additional Context:** The `replit.md` documented that the production build should run `drizzle-kit push --force` before compiling, but this had never been implemented in the actual `artifact.toml` build configuration.

**Fix Applied:** Updated `artifacts/api-server/.replit-artifact/artifact.toml` production build step from:
```toml
[services.production.build]
args = ["pnpm", "--filter", "@workspace/api-server", "run", "build"]
```
To:
```toml
[services.production.build]
args = ["sh", "-c", "pnpm --filter @workspace/db run push-force && pnpm --filter @workspace/api-server run build"]
```

This ensures every publish automatically syncs the Drizzle schema to the production database before compiling the server bundle.

**Resolution:** The user resolved the immediate Provisioning blockage by removing the problematic key. A subsequent publish succeeded — production health endpoint (`https://textitie.replit.app/api/healthz`) now returns `{"status":"ok"}`.

**Note:** The successful publish may have occurred before the `artifact.toml` fix was committed, meaning the schema sync step may not have run during that particular deploy. The fix will take effect on the next publish.

## Open Items & Recommendations

### DATABASE_URL: Dev vs Production Separation

**Current State:** `DATABASE_URL` is a shared Replit secret — the same value is available in both dev and production environments. Dev and production use separate PostgreSQL databases (confirmed by the user), but the Replit configuration does not currently enforce this separation.

**The Problem:** Replit secrets are global and cannot have different values per environment. Only environment-scoped env vars support different values for development vs production. This means:
- The production build's schema sync (`drizzle-kit push --force`) will target whichever database the shared `DATABASE_URL` points to
- There is no guardrail preventing accidental schema pushes to the wrong database

**Recommendation:** Split `DATABASE_URL` into environment-scoped variables:
1. Delete the shared `DATABASE_URL` secret
2. Set `DATABASE_URL` as a **development** env var (dev database connection string)
3. Set `DATABASE_URL` as a **production** env var (production database connection string)

This ensures the schema sync and runtime both target the correct database automatically based on the environment. The trade-off is that env vars are visible in the Replit GUI (unlike secrets), but it is the only mechanism Replit provides for environment-specific values.
