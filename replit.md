# Textitie — Two-way SMS Platform (codename SAMA)

## Overview
Textitie (internal codename "SAMA" — Simple but Advanced Messaging Alternative) is a multi-tenant, compliance-first conversational SMS platform. Tenants get a Textline-style agent inbox; the platform operator gets a Master Conductor (the "SAMA Control Plane") for tenant management, message injection, phone-number provisioning, and webhook monitoring. Intelligence comes from a per-tenant Grok-powered knowledge pipeline (Knowledge Base → Library → Professor → Classroom → Student).

## System Architecture

### Core
-   **Monorepo**: pnpm workspace (`artifacts/*` deployables, `lib/*` shared libs, `scripts/`).
-   **Contract-first API**: OpenAPI (`lib/api-spec/openapi.yaml`) generates the typed React Query client (`lib/api-client-react`) + Zod schemas (`lib/api-zod`). Regenerate with `pnpm --filter @workspace/api-spec run codegen`. Use **verb-first operationIds** to avoid Orval symbol mangling.
-   **API server**: Express.js + TypeScript (`artifacts/api-server`); one router mounted under `/api` (`src/routes/index.ts`). Conductor routes use HTTP Basic; tenant routes use `requireTenantAuth` (JWT). Never `console.log` in server code — use `req.log` / the singleton `logger`.
-   **Admin UI**: React + Vite + wouter + shadcn (`artifacts/eng-architect`), served at `/admin/`.
-   **User UI**: React + Vite + wouter + shadcn (`artifacts/user-app`). Public Landing at `/`; agent Inbox at `/inbox`. Other auth-gated routes: `/contacts`, `/settings`, `/billing`, `/automations`, `/campaigns`, `/analytics`, `/knowledge`, `/profile`, and the `/onboarding/*` flow. Public: `/login`, `/verify`, `/signup`, `/signup/trial`, `/privacy`, `/terms`.
-   **Tenant lifecycle**: reversible soft-archive (`tenants.lifecycle_status` = `active|archived`, with `archivedAt/By`, `archiveReason`, `purgeAfter`, `purgeBlockedReason`) + Conductor unassign/return-number-to-pool + a 60s-timer scheduled **hard-purge** that fires 30d after archive **only when the tenant owns 0 phone numbers** (`phone_numbers` FK is **RESTRICT**), plus the manual slug-confirmed DELETE. **Pool auto-assign/release**: a trial **signup** auto-claims one number from the Admin → Phone "Available" pool (= owned-by-Twilio minus the registry, computed exactly like `routes/telephony.ts`) onto the new tenant's **Demo Department** — best-effort, never throws, gated on `getPublicWebhookConfig().available` so dev/preview never consumes a real number (`lib/phonePool.ts` `claimPoolNumberForDepartment`, wired post-commit in `routes/tenantAuth.ts`). **Archive** auto-releases ALL the tenant's numbers back to the pool (`releaseAllTenantNumbers` in `phoneNumberRegistry.ts`, best-effort in the archive route), which also unblocks the purge. **Trial expiry deliberately KEEPS the demo number** ("your demo number and setup are saved") — release is on archive, NOT expiry. The purge locks + re-checks eligibility inside the same transaction as the delete (closes the restore-vs-purge race). Archived tenants are blocked from login, inbound, and **all backend-scheduled outbound** (timer skips them; `isTenantSendingExpired` returns true before the billingBypass unlock). Protected seed slugs (`acme`) refuse archive (400) / delete (403). Code: `lib/tenantLifecycle.ts`, `lib/timerEngine.ts`, `lib/demoTextingGate.ts`, routes in `routes/tenants.ts` (archive/restore/unassign/DELETE); Admin UI in `eng-architect` Tenants + TenantDetail.

### Database
-   **PostgreSQL + Drizzle ORM.** All data lives in the `public` schema with explicit `tenant_id` scoping on every per-tenant table (`conversations`, `opt_outs`, `automation_rules`, `campaigns`, `reminders`, `audit_logs`, knowledge tables, …); `messages` inherits scoping via `conversation_id`.
-   **DO NOT add a DB CHECK / enum constraint for free-form classification columns** (e.g. `category`, `region`): a single invalid row 500s the whole list endpoint when the route returns raw Drizzle rows. Use a plain text column + app-level validation + a safe fallback.
-   **Database environments & schema migration**: Replit **managed Postgres** (`DATABASE_URL` is runtime-managed); development and production are **separate** managed databases. Production schema is updated **automatically by the Publish flow** — on publish, Replit diffs the dev schema against prod and applies it (renames/destructive alters prompt for confirmation in the Publish UI). There is **no** custom migration step, deploy-build hook, or startup DDL. To ship a schema change to prod: edit the Drizzle schema → push to dev (`pnpm --filter @workspace/db run push-force`) → re-publish. Publish migrates **schema, not data**. The agent has **read-only** access to prod (SELECT only); prod data writes go through the Conductor API, never raw SQL.

### LLM knowledge pipeline (Grok / xAI)
Per-tenant flow **Knowledge Base → Library → Professor → Classroom → Student**, split across two providers (both via the OpenAI-compatible chat-completions API; clients in `artifacts/api-server/src/lib/grokClient.ts`). The reasoning **Professor** (knowledge curation, Library conflict adjudication) is a **creation-only** tool — Human + Professor via the Conductor; **it is no longer on any live inbound path** (removed from the Co-Pilot runtime 2026-06-27; Auto-Pilot was already closed-book). It runs on **OpenRouter (Qwen)** through the **Replit AI Integrations proxy** — no key of ours, billed to Replit credits; default model `qwen/qwen3-max` (a fast non-thinking tier; override `SAMA_PROFESSOR_MODEL`); client `professorClient()` / guard `professorConfigured()` keyed on `AI_INTEGRATIONS_OPENROUTER_BASE_URL` + `_API_KEY`. The fast **Student** (inbound draft replies) stays on **Grok (xAI)** (`baseURL https://api.x.ai/v1`, key in the `GROK_KEYS` secret); model `grok-4.20-0309-non-reasoning` (override `SAMA_STUDENT_MODEL`); client `grokClient()` / guard `grokConfigured()`. **Each role degrades to a stub when its own provider is unconfigured** so inbound SMS never breaks. Postgres full-text retrieval only (tsvector + GIN) — **no vectors**.

Key files: `artifacts/api-server/src/lib/knowledge.ts` (Library, extraction, `retrieveClassroomFacts`, `classifyQueryCategory`, `hasUnresolvedConflicts`), `lib/librarian.ts` (push-time dedup/conflict adjudication), `routes/knowledge.ts` (Conductor + tenant routes), `routes/webhooks.ts` (Student pipeline), `lib/ai-student/src/index.ts` (Student prompt + `parseStudentSections`), `lib/engagementPolicy.ts` (`evaluateAutoPilotTurn`), `lib/outboundReply.ts` (`sendConversationReply`), `eng-architect/src/pages/Professor.tsx` (Professor UI). Schema: `lib/db/src/schema/knowledge.ts`.

Agent-critical behavior (full detail in README):
-   **Facts** carry a `category` (`pricing|compliance|features|technical_setup|general`, app-validated, `general` fallback) and a status (`draft` → `published`, plus `conflict`). The Facts GET returns **raw Drizzle rows with no mapper** — any new column must also be added (required, correct nullability) to the OpenAPI schema or the generated UI type won't expose it.
-   **Engagement mode (three canonical modes)**: per-tenant `tenants.engagementMode` is one of `manual | copilot | autopilot` (**DB default `copilot`**), with an optional **per-conversation override** `conversations.engagementModeOverride` (`null` = inherit the tenant default). The effective mode = override ?? tenant mode (`resolveEffectiveEngagementMode`). Legacy values are **aliased on write and normalized** (`assisted`→`copilot`, `gated_auto`→`autopilot`) — **no data migration**; canonical list + normalization live in `lib/engagementPolicy.ts` (`ENGAGEMENT_MODES`, `normalizeEngagementMode`, unknown→`copilot`). Behavior per mode:
    -   **Manual** (🔵 Blue): AI fully off — no Student draft, no auto-send, no learning. Inbound supersedes/clears any pending AI state.
    -   **Co-Pilot** (🟡 Yellow): the **Student (Grok)** drafts — grounded in the approved Classroom when it matches, otherwise its own best draft (or the per-tenant fallback phrase). **The Professor is NOT consulted at runtime** (its live escalation was removed 2026-06-27). The draft is upserted to `conversation_ai_states` (status `drafted`) and pre-filled into the inbox composer; a human edits + sends. **NEVER learns** (no fact persistence).
    -   **Auto-Pilot** (🟢 Green): **closed-book, fail-OPEN responder + fallback circuit breaker** — no Professor escalation, no `persistEscalatedFacts`/`retrieveLibraryContext`, **NEVER learns**. Per inbound (first match wins): a knowledge match in the approved Classroom → stitch a **grounded** Student answer, auto-send, stay green (resets the consecutive-fallback run); **no match → graceful out-of-scope ack and the conversation CONTINUES green**; responder/LLM error → graceful fallback ack (**never silent**) that counts toward the breaker. **Breaker**: on the 3rd *consecutive* fallback **OR** >3 fallbacks in a rolling 2-min window → send a final ack and **step the conversation down GREEN→BLUE** (`conversations.engagementModeOverride='manual'`, **not** auto-cleared — a human re-enables once they've trained it). Compliance/opt-out are **hard** (re-checked at send); a human can step in any time. Logic lives in the pure gate table `evaluateAutoPilotTurn` + the `autopilot_turn_events` breaker history; runner `runAutoPilotFailOpenTurn` (`lib/inboundAiPipeline.ts`).
-   **Learning rule**: **NO engagement mode learns at runtime.** Auto-Pilot is closed-book (no Professor, no persistence) and **Co-Pilot's live Professor-escalation self-learning loop was REMOVED 2026-06-27** — Co-Pilot now drafts with the Student (Grok) only. The Professor is a **creation-only** tool (Human + Professor via the Conductor), so the only way facts enter the Classroom is a human-driven push. `manual` skips AI entirely. No inbound path persists facts.
-   **AI state**: one row per conversation in `conversation_ai_states` (tenant-scoped, upsert/supersede in `lib/aiStateStore.ts`); statuses `idle|drafted|auto_sent|failed|refused|human_handled|superseded`. The conversations list/detail return `effectiveEngagementMode` + `aiState`; `PATCH /conversations/:id` accepts a validated, tenant-scoped `engagementModeOverride` (aliases folded). A **human send** marks the pending `drafted`/`refused`/`failed` row `human_handled` (no learning), so an Auto-Pilot conversation returns to green afterward. Inbox send-button color is derived live from effective mode + AI state.
-   **Auto-send gate**: `autopilot` now runs the **fail-OPEN** closed-book gate table `evaluateAutoPilotTurn` (match → grounded answer; no match / responder error → graceful ack; breaker → final ack + stepdown to `manual`), backed by the append-only `autopilot_turn_events` history (`recordAutopilotTurnEvent` / `getAutopilotFallbackCounts`, **idempotent on `(tenant_id, inbound_message_id)`** so a carrier retry never double-counts the breaker; counting rules in the pure `computeAutoPilotFallbackCounts`). The legacy **fail-closed** `evaluateAutoSend` / `evaluateProfessorEscalationSend` gates and the unreachable legacy Auto-Pilot branch that called them were **deleted 2026-06-27** (along with the live Professor escalation path) — the only live auto-send path is the fail-OPEN gate table above. Compliance is **re-checked at send time** in `sendConversationReply`. Auto-send is **idempotent** via `ai_auto_replies` (unique `(tenant_id, inbound_sid)`); a send that **fails or throws releases the claim AND writes no turn event** (so the breaker only advances on a confirmed send) and a webhook retry can re-attempt. The whole pipeline runs **off the inbound 200 path** via the durable per-conversation FIFO worker, after the automation engine (gated on `!result.handled`).
-   **Live escalation / self-learning loop — REMOVED from the runtime path (2026-06-27).** Previously a Co-Pilot ungrounded inbound (`!kbMatched`) escalated, off the 200 path, to the autonomous **Professor** for a polished customer reply + auto-persisted facts. The measured cost was ~full reasoning-LLM latency on every ungrounded turn for what amounted to draft polish — the Professor tax — while it could also auto-publish self-attested facts. **It is gone:** the Professor is now **creation-only** (Human + Professor via the Conductor). A Co-Pilot ungrounded tenant-specific inbound now simply keeps the **Student's (Grok) own draft** (or the per-tenant fallback phrase) for a human to review in the composer — faster, cheaper, and no autonomous fact persistence. The runtime functions `professorEscalate` / `screenEscalatedFacts` / `persistEscalatedFacts` and the deterministic injection screens still **exist** (used by the Conductor creation flow) but **nothing on the inbound path calls them**; the legacy auto-send gate `evaluateProfessorEscalationSend` and the dead Auto-Pilot branch that referenced them were **deleted 2026-06-27**. The durable per-conversation FIFO staging queue (`conversation_inbound_ai_stages`) remains and still serializes the Co-Pilot draft pipeline. Key code: `lib/inboundAiPipeline.ts` (Co-Pilot branch), `lib/knowledge.ts`, `routes/webhooks.ts`.
-   **Brain knowledge pull (Conductor manual sync)**: a Conductor-only avenue harvests knowledge from the external **Brain/Beast** service into a tenant's pipeline — **"Brain + Human" MIRRORS "Human + Professor."** Approved Brain content lands in the SAME `absorbed_facts` pool (`source='brain'`, `sessionId=null`, `sourceUrl` for provenance) and rides the SAME full-snapshot Classroom push (Librarian adjudication), so it is Student-groundable for free with **no separate Brain table/push**. Flow: `POST /tenants/:id/brain/pull` (synchronous harvest → staged `draft`; near-dup statements deduped; contradictions staged `conflict` carrying a flag reason in `conflictReason`) → `GET …/brain/candidates` → human approves (editable category, flagged candidates render unchecked) → `POST …/brain/push`. The push **validates every selected id is a tenant-scoped `source='brain'` actionable candidate and 400s if any isn't (no partial promote)**, then promotes to `published` and snapshots. Degrades to **503** until `BRAIN_BASE_URL` + `BRAIN_API_KEY` are set; the wire contract in `brainClient.ts` is **assumed/tolerant** pending a real endpoint. **Invariant: the Classroom push is ALWAYS a full UNION of every published absorbed fact (Professor + Brain) — the legacy per-session *subset* push was removed so a session push can never wipe Brain facts.** Conductor-auth, `tenant_id`-siloed, cron parked. Key code: `lib/brainClient.ts`, `lib/classroomPublish.ts` (shared snapshot helper owning the advisory lock + supersede + adjudication), `routes/brain.ts`, `eng-architect/src/pages/Brain.tsx`.
-   **Backward compatibility**: the legacy `tenants.knowledge_base` text column is **retained** as the Student's fallback when a tenant has no published Classroom. The tenant `/knowledge` page and its upload endpoint still exist, and the inbox **"Halo AI" button is still a placeholder** pending wiring to the live Student draft API.

## Stage 4 Multi-Tenancy Migration (DEFERRED)
Schema-per-tenant isolation was rolled back at the data layer: webhook writes went to `tenant_<slug>.*` while the inbox kept reading `public.*`, so auto-replies, opt-outs, and campaign attribution silently disappeared wherever the tenant schema actually existed. Prod "worked" only because it never had `tenant_<slug>` schemas (so `search_path` fell through to `public`); dev kept breaking because the schemas existed.

**Current state:** `lib/db/src/tenant-db.ts` `getTenantDb(slug)` / `getTenantPool(slug)` are thin wrappers that return the **global pool**, so call sites don't change when isolation is re-enabled. All read/write paths use `public.*` scoped by `tenant_id`. The slug param is preserved + validated so per-tenant routing can be re-enabled by changing only `tenant-db.ts`.

**Re-enablement (only when isolation is actually required — SOC2/HIPAA/sovereign-data):** restore the `search_path` logic in `tenant-db.ts`; re-run `scripts/src/provision-tenant-schemas.ts` + `backfill-tenant-schemas.ts`; switch the inbox read path (`routes/conversations.ts`, `routes/messages.ts`) to `getTenantDb(slug)`; drop the now-redundant `tenant_id` filters in queries that move into a tenant schema. Original plumbing (provisioner, template SQL, backfill scripts, pre-rollback pg_dump backups in `John/`) is still on disk, unused. The dormant `tenant_*` schemas in dev hold a pre-rollback snapshot and can be dropped anytime.
